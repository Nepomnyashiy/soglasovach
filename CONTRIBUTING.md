# Руководство для Разработчика (CONTRIBUTING.md)

Этот документ содержит рекомендации и инструкции для разработчиков, желающих внести свой вклад в проект "Согласовач".

## 1. Общие Принципы

*   **Чистый Код:** Мы придерживаемся принципов чистого кода (Clean Code) и стремимся к читаемому, поддерживаемому и тестируемому коду.
*   **Тестирование:** Все новые функции и исправления ошибок должны сопровождаться соответствующими тестами.
*   **Code Review:** Все изменения проходят обязательное ревью кода.

## 2. Локальная Разработка

Убедитесь, что у вас запущены все сервисы через Docker Compose, как описано в `README.md`.

### 2.1. Запуск Тестов (Backend)

Для запуска тестов бэкенда внутри Docker-контейнера:

```bash
docker compose exec backend pytest
```

### 2.2. Проверка Качества Кода (Backend)

Мы используем `ruff` для линтинга и форматирования, а также `mypy` для статической проверки типов.

*   **Линтинг и форматирование с `ruff`:**
    ```bash
    docker compose exec backend ruff check .
    docker compose exec backend ruff format .
    ```
*   **Статическая проверка типов с `mypy`:**
    ```bash
    docker compose exec backend mypy .
    ```

### 2.3. Работа с Базой Данных (Миграции Alembic)

Мы используем Alembic для управления миграциями базы данных.

*   **Создание новой миграции:**
    ```bash
    docker compose exec backend alembic revision --autogenerate -m "Описание вашей миграции"
    ```
    После создания файла миграции, отредактируйте его вручную, если `autogenerate` не справился со всеми изменениями.
*   **Применение миграций:**
    ```bash
    docker compose exec backend alembic upgrade head
    ```
*   **Откат миграций:**
    ```bash
    docker compose exec backend alembic downgrade -1
    ```

## 3. Внесение Изменений

1.  Создайте новую ветку для вашей задачи (`git checkout -b feature/your-feature-name`).
2.  Внесите изменения, напишите тесты.
3.  Убедитесь, что все тесты проходят, и линтеры не выдают ошибок.
4.  Сделайте коммит (`git commit -m "feat: Описание вашей фичи"`).
5.  Отправьте ветку в удаленный репозиторий (`git push origin feature/your-feature-name`).
6.  Создайте Pull Request.
